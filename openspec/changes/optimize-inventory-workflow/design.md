## Context

当前盘点模块已实现完整的 6 状态流转（待盘点→盘点中→待审核→已完成/已驳回/已作废）、盘点执行（扫码/手动录入）、漏盘规则、重复盘点规则、多人协作、库存自动调整。但缺少盘点期间的库存锁定机制——盘点进行中时，同一分公司的资产仍可被采购入库、领用出库、调拨等操作修改数量，导致盘点基准数据不一致。

后端使用 Django REST Framework，Transfer 的 6 种操作（purchase/assign/return/transfer/repair/scrap）通过 `TransfeViewSet` 的独立 action 端点创建。Asset 模型的 `数量` 字段和 `分公司编号` 字段是锁定校验的关键字段。当前无任何 `is_locked`、`frozen` 等锁定标记。

## Goals / Non-Goals

**Goals:**
- 盘点开始后自动锁定该分公司库存，阻止流转操作修改资产数量
- 审核通过或作废后自动解锁
- 同一分公司同时只允许一个活跃盘点任务
- 驳回后支持选择性重盘（仅重置异常项）
- 盘点报告增加差异率和分类汇总

**Non-Goals:**
- 不实现细粒度的单资产锁定（仅按分公司维度锁定）
- 不修改 Asset 模型结构（不新增锁定字段）
- 不实现定时自动盘点或盘点计划调度
- 不修改移动端盘点扫码的交互逻辑

## Decisions

### 1. 锁定机制：动态查询 vs 模型字段

**选择**: 动态查询方式——在 Transfer 创建时查询该分公司是否存在活跃盘点任务
**理由**: 无需数据库迁移，无需维护锁定状态同步（盘点作废/完成时忘记解锁的风险），逻辑集中在查询层面
**备选**: 在 Branch 模型上增加 `locked_by_inventory` FK 字段——需要迁移，且需在多个地方维护状态一致性

### 2. 锁定范围

**选择**: 按分公司维度锁定，锁定范围为 status in ['in_progress', 'pending_review'] 的盘点任务所关联的分公司
**理由**: 与业务需求一致——盘点是对整个分公司或某分类的全部资产进行盘点，锁定粒度应与盘点粒度匹配
**备选**: 按单资产锁定——实现复杂，且当前盘点任务本身就以分公司为单位生成盘点项

### 3. 流转拦截方式

**选择**: 在 TransferViewSet 的 `perform_create` 层面进行统一拦截校验
**理由**: 所有 6 种流转操作都经过 `perform_create`，只需在一处添加校验逻辑，避免在每个 action 中重复代码
**备选**: 在每个 action 中单独校验——代码重复，容易遗漏

### 4. 并发盘点限制

**选择**: 在 `start` action 中校验同一分公司是否已有 status in ['in_progress', 'pending_review'] 的任务
**理由**: 在盘点开始时拦截而非创建时拦截，允许提前创建多个待盘点任务但只允许执行一个
**备选**: 在创建任务时即拦截——过于严格，用户可能需要提前规划多个盘点任务

### 5. 选择性重盘实现

**选择**: `recount` action 增加 `reset_scope` 参数（`all` | `abnormal_only`），默认 `all` 保持向后兼容。`abnormal_only` 模式仅重置 result in ['surplus', 'missing', 'unchecked'] 的盘点项
**理由**: 大多数驳回场景中只有少量异常项需要重盘，保留正常项可大幅提升效率
**备选**: 始终全部重置——简单但浪费人力，与用户需求不符

### 6. 盘点报告增强

**选择**: 在现有 report endpoint 返回数据中增加 `summary` 对象，包含差异率、分类汇总、变动统计；Item 序列化器增加 `quantity_before` 和 `quantity_after` 字段
**理由**: 最小化 API 变更，向后兼容，前端可渐进式展示新增字段
**备选**: 新增独立的 report-summary endpoint——增加复杂度且不必要

## Risks / Trade-offs

- **[风险]** 动态查询在高并发场景下可能有性能开销，每次创建流转都需查询 inventories 表 -> 缓解: 在 `start` 时在 InventoryTask 表的 `(branch_id, status)` 上建立联合索引，查询开销极小
- **[风险]** 如果分公司维度锁定的盘点任务 category 不为空（仅盘点某分类），则仍会锁定整个分公司的所有流转操作 -> 缓解: 首期按分公司锁定，后续可按 branch+category 细化锁定范围
- **[权衡]** 选择性重盘增加了 recount action 的复杂度，但换取了显著的效率提升，在驳回场景中用户体验更好
- **[权衡]** 不锁定分类维度的盘点——如果用户仅盘点"电子设备"类，当前设计仍会锁定整个分公司的流转操作。这是保守策略，避免漏锁导致数据不一致
