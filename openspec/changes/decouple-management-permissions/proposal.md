## Why

当前系统把**职位**与**管理权限**耦合在同一个 `role` 字段上（admin/manager/supervisor/leader/staff 五选一）。`role` 既表示员工在组织架构中的职位归属，又直接决定其数据管理范围（`core/permissions.py` 的 `DataScopeMixin` 完全由 `role + region/branch` 外键推导），且接口级权限（`IsRoleMin` / `min_role` / `CanApprove`）也硬绑在角色等级上。

这带来两个无法用现状解决的真实痛点：

1. **行政主管同时管理自己的行政组**：一位 `supervisor`（行政主管）的 `DataScopeMixin` 范围是"本大区"，但他若同时需要**精细地只管理自己带的某个行政组**、或**跨大区管理某些团队**，现有模型无能为力——范围完全由职位推导，无法单独授予。
2. **授权不灵活**：无法给"不同员工单独添加不同的管理权限"。要调整任何人的管理范围，只能改其 `role`（连职位带组织归属一起变）或挪动其 `region/branch` 外键，牵一发动全身。`leader`（组长）角色甚至名不副实——数据范围逻辑里 leader 与 staff 同走"本分公司"分支，看不到下属。

根本原因：**没有独立的授权实体**。权限是代码常量 + if/elif 硬编码，扩展新范围需改源码。需要解耦出独立的管理权限模型。

## What Changes

- **解耦两套体系**：
  - **职位/组织归属**（原有 `role`）：仅表示员工在组织架构中的位置，用于归属与展示，**不再直接决定数据管理范围**。
  - **管理权限**（新增）：独立授权模型，可给任意员工单独授予具体的管理权限。
- **新增管理授权模型**：支持**两者结合**的授权粒度——
  - **组织节点维度**：授权管理某些大区 / 分公司 / 行政组（数据范围授权）。
  - **业务操作维度**：授权某些业务模块 / 操作（功能权限授权，如审批采购单、管理盘点）。
- **`DataScopeMixin` 重构**：数据范围不再由 `role` 推导，改为查询员工被授予的组织节点范围（admin 例外，仍为全部）。
- **接口级权限解耦**：`IsRoleMin` / `min_role` / `CanApprove` 等改为基于"是否拥有某业务操作权限"判断，而非角色等级。
- **admin 例外**：超级管理员仍由职位决定，拥有全部权限，不进入新授权模型（避免误操作锁死系统）。
- **管理后台**：提供"为员工分配管理权限"的界面（选择员工 → 勾选组织节点 + 业务操作）。

## Capabilities

### New Capabilities
- `management-permissions`: 独立的管理授权模型，支持按组织节点（大区/分公司/行政组）与业务操作（模块/动作）两个维度为员工授予管理权限；admin 走职位最高权限，不参与授权。

## Impact

- **后端模型（新增）**：`apps/users` 或新建 `apps/permissions` 下新增管理授权模型（组织节点授权 + 业务操作授权，与 User 为 M2M 或外键关系）。
- **后端权限层（重构）**：`core/permissions.py` 的 `DataScopeMixin`（数据范围改查授权）、`IsRoleMin` / `CanApprove`（接口级改查业务操作权限）。
- **后端接口（新增）**：管理授权的 CRUD 接口（分配 / 查询 / 撤销员工权限），及其在 `apps/users/views.py` 或新 app 下的挂载。
- **数据迁移**：提供从现有 `role + region/branch` 推导的**默认授权迁移**（如 supervisor 默认授其所在大区管理权），保证平滑过渡、不丢失既有能力。
- **前端**：新增"管理权限分配"页面；前端 `hooks/usePermission.ts` 与 `store/user.ts` 的权限判断从单一 `role` 改为消费用户的管理权限集。
- **回归验收**：admin 仍全权；原有各职位用户的既有管理能力经迁移后保留；行政主管可被单独授予"仅管理某行政组"或"跨大区管理"且数据范围正确收窄/扩大；无授权的普通员工仅见自身相关数据。
- **风险**：权限模型重构触及全站数据隔离，需充分的回归测试与数据迁移验证。
