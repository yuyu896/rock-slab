## Context

当前前端项目已完成 UI 框架搭建，包含 10+ 个视图页面、完整 API 层和 Pinia 状态管理。但功能验证发现大量页面仅为 UI 外壳：按钮无事件绑定、核心表单缺失、API 已定义但未在视图中引入。后端 API 和数据模型基本完备，问题集中在前端视图层的业务逻辑串联。

技术栈：Vue 3 + TypeScript + Element Plus + Pinia + Vite，后端 Django REST Framework。

## Goals / Non-Goals

**Goals:**

- 将所有占位符/空壳功能替换为可工作的完整实现
- 打通资产流转 5 种单据的创建→审批→驳回全流程
- 打通盘点任务从创建到完成的全生命周期
- 补全缺失的关键页面：资产详情、组织架构图、通知列表
- 修复失效的交互：搜索筛选、状态切换、角色权限视图

**Non-Goals:**

- 不修改后端 API 接口设计（前端适配现有后端）
- 不引入新的 UI 框架或重构现有组件结构
- 不做国际化（i18n）
- 不做性能优化（虚拟滚动、懒加载等）
- 移动端不做原生 App，仅 Web 响应式 + PWA
- 暂不实现 WebSocket 实时推送（使用轮询替代）

## Decisions

### 1. 资产流转：Tab 式统一视图 vs 独立页面

**选择**：在现有 Transfer.vue 中采用 Tab 页签切换，分为"全部"、"领用出库"、"归还"、"调拨"、"维修"、"报废"六个 Tab。

**理由**：Transfer.vue 已有列表展示和审批功能，新增 Tab 切换成本最低。每种类型共用列表框架，仅表单弹窗不同。独立页面会导致导航复杂度增加。

**替代方案**：每种流转类型一个独立页面 → 拒绝，导航过于分散，用户需要在多个页面间切换。

### 2. 资产详情：Drawer 抽屉 vs 独立路由页面

**选择**：使用 Element Plus 的 `el-drawer` 右侧抽屉展示详情，不新增路由。

**理由**：资产列表是高频操作页，用户查看详情后通常回到列表继续操作。抽屉可以保留列表上下文，减少页面跳转。移动端则跳转到已有的 AssetDetail.vue 路由。

### 3. 盘点执行：在现有 Inventory.vue 的三视图架构内补全

**选择**：沿用 Inventory.vue 已有的列表/详情/扫描三视图，补全缺失的事件绑定和 API 调用。

**理由**：三视图架构已设计好，UI 布局完整，只需补全逻辑。避免大规模重写。

### 4. 条码生成：JsBarcode 库

**选择**：使用 JsBarcode 生成 Code128 一维码，渲染到 Canvas/SVG。

**理由**：轻量（~40KB）、纯前端生成、支持 Code128 格式、可打印。无需后端参与。

**替代方案**：后端生成图片 → 拒绝，增加后端复杂度且无必要。

### 5. 组织架构图：纯 CSS + Vue 递归组件

**选择**：使用 Vue 3 递归组件 + CSS Flexbox 实现树形组织架构图，不引入第三方图表库。

**理由**：组织架构图结构简单（集团→区域→分公司→人员），用递归组件即可实现。避免引入 d3-org-chart 等重型依赖。Element Plus 的 `el-tree` 可作为降级方案。

### 6. 分类动态属性：前端 JSON Schema 驱动

**选择**：在分类编辑表单中增加"属性模板"JSON 字段，前端根据 JSON Schema 动态渲染属性输入框。

**理由**：后端 Category 模型暂不修改属性结构，前端先用 JSON Schema 方案在资产创建/编辑表单中动态展示分类相关属性。后端可在后续迭代中补充专门的属性模型。

**替代方案**：后端新增 CategoryAttribute 模型 → 暂缓，当前阶段前端先行。

### 7. Excel 导入导出：沿用后端 API

**选择**：分类导入新增 API 端点 `POST /api/categories/import`，采购导入使用已有 `POST /api/assets/import`。

**理由**：资产导入后端已实现（openpyxl），分类导入需要新增后端端点。前端统一使用 FormData 上传文件。

### 8. 移动端扫码：getUserMedia + BarcodeDetector API

**选择**：使用浏览器原生 `BarcodeDetector` API（Chrome 88+），降级方案为手动输入。

**理由**：BarcodeDetector 是 Web 标准 API，无需引入第三方库。不支持的环境自动降级为手动输入模式。

### 9. 角色权限视图：前端过滤 + 后端参数

**选择**：Dashboard 和报表 API 请求时传递当前用户的角色和关联 ID（branchId/regionId），后端返回范围数据。前端不做二次过滤。

**理由**：安全性要求权限控制在后端实施。前端仅需传递参数和展示后端返回的数据。

## Risks / Trade-offs

- **[风险] 盘点扫描逻辑复杂** → 缓解：MobileScan.vue 已有确认模态框和 API 调用框架，补全事件绑定为主
- **[风险] 分类动态属性后端未支持** → 缓解：前端先用 JSON Schema 存储，数据存于分类的备注字段或前端本地，后续迁移到后端
- **[风险] BarcodeDetector 浏览器兼容性** → 缓解：不支持时自动降级为手动输入，不影响核心功能
- **[权衡] 组织架构图使用递归组件**：视觉效果不如专业图表库，但足够满足 MVP 需求
- **[权衡] 统一 Tab 式流转视图**：单页面代码量较大，需注意组件拆分避免单文件过长
