## Context

分公司编号 `Branch.code` 是系统中 Branch 模型与 `Asset.分公司编号` CharField 之间的关联键。盘点模块通过 `分公司编号=task.branch.code` 筛选资产，报表模块通过 `分公司编号__in=branch_codes` 隔离数据，采购入库时将 `branch.code` 写入新资产的 `分公司编号`。

当前 code 字段仅有 `unique=True` 约束，无格式校验。seed_data.py 中已有数据遵循 `SH001`/`HZ001`/`GZ001` 模式（2-4位城市拼音缩写+3位数字）。

## Goals / Non-Goals

**Goals:**
- 建立统一的分公司编号格式规则：`{城市缩写}{DDD}`
- 后端模型层和序列化器层双重校验
- 前端表单实时校验与自动格式化（大写转换）
- 与现有数据兼容（SH001/HZ001/GZ001 已符合）

**Non-Goals:**
- 不改动 Asset.分公司编号 字段（那是历史数据字段，格式由 Branch.code 决定）
- 不自动生成编号（编号由用户手动指定，保持语义化）
- 不做编号变更的历史数据迁移（编号变更属于业务决策）

## Decisions

### 1. 格式正则：`^[A-Z]{2,4}[0-9]{3}$`

**选择理由**: 2-4位字母覆盖中国城市名缩写（SH/BJ/GZ/HRB 等），3位数字支持每个城市最多999个分公司。

**替代方案**: 纯数字 `^[0-9]{4}$` — 无语义信息，被否决。

### 2. 校验层次：模型 validator + 序列化器 validate_code

**选择理由**: 模型层 validator 是数据库安全网（即使绕过 API 直接操作模型也会校验），序列化器层做额外处理（trim、大写转换）。

### 3. 前端自动大写

**选择理由**: 用户输入 `sh001` 时自动转为 `SH001`，减少出错。通过 `input` 事件监听实现，比 CSS `text-transform` 更可靠（CSS 只改显示不改值）。

## Risks / Trade-offs

- **[现有不合规数据]** → 迁移脚本检查并修复。当前 seed_data 已合规，风险低。
- **[编号长度限制]** → 3位数字上限999，对中小型组织足够。如需扩展可改为4位。
- **[用户习惯变更]** → placeholder 和错误提示明确说明格式要求。
